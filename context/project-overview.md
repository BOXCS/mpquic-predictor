# MP-QUIC LSTM IoT — Network Quality Degradation Prediction System

## Overview

This system predicts IoT network quality degradation using an LSTM model
and proactively triggers path switching on a Multipath QUIC (MP-QUIC)
connection before degradation impacts data delivery. A Raspberry Pi 5
reads temperature and humidity data from a DHT22 sensor and transmits
it simultaneously over two network paths (WiFi and LAN). A server-side
LSTM model continuously monitors network metrics (RTT, goodput) and
issues a switching recommendation when degradation is predicted, allowing
the system to reroute traffic to the healthier path before quality drops.
A real-time web dashboard visualizes path status, prediction results,
and historical metrics for monitoring and thesis evaluation purposes.

## Goals

1. Predict network quality degradation at least one step ahead using
   an LSTM model trained on 14 simulated degradation scenarios
2. Trigger proactive path switching on MP-QUIC before degradation
   affects IoT data delivery, reducing packet loss and latency spikes
3. Demonstrate measurable improvement over a Round Robin baseline
   in terms of RTT stability and goodput under degradation conditions
4. Provide a real-time dashboard for monitoring path status,
   predictions, and switching events during lab evaluation
5. Produce evaluation artifacts (charts, logs, metrics) that are
   directly usable in the thesis report

## Core System Flow

1. Raspberry Pi reads temperature and humidity from DHT22 sensor
   at a configured interval
2. MP-QUIC client sends sensor data simultaneously over wlan0 (WiFi)
   and eth0 (LAN) to the server
3. Server receives data and computes per-path metrics:
   RTT and goodput for each active path
4. LSTM model runs inference on the latest metric window and outputs
   a degradation prediction for each path
5. If degradation is predicted on the active path, the server sends
   a switching recommendation to the Raspberry Pi
6. Raspberry Pi executes path switching via `path_switcher.py`,
   rerouting traffic to the healthier path
7. All metrics, predictions, and switching events are written
   to SQLite by `db_writer.py`
8. FastAPI backend exposes the latest data via REST and WebSocket;
   React dashboard updates in real time without page refresh

## Features

### LSTM-Based Degradation Prediction
- Trained on 14 network degradation scenarios generated by the simulator
- Predicts degradation one step ahead using a sliding window of
  RTT and goodput measurements
- Runs server-side; TFLite fallback available on Raspberry Pi
  if server is unreachable

### Proactive MP-QUIC Path Switching
- Maintains two simultaneous paths: wlan0 (WiFi) and eth0 (LAN)
- Switching is triggered by LSTM prediction, not reactive packet loss
- Path switching recommendation is sent from server to Raspi
  and executed by `path_switcher.py`

### Real-Time Web Dashboard
- Displays live RTT and goodput charts per path (Recharts)
- Shows current path status, active path indicator, and alert banner
  when a switching event occurs
- Updates via WebSocket — no polling, no page refresh required
- History page shows past metrics and switching events from SQLite

### Evaluation & Comparison
- Compares LSTM-based switching vs Round Robin baseline
- Tests early warning accuracy against ground-truth degradation labels
- Generates publication-ready charts for the thesis report

### Sensor Data Logging
- DHT22 temperature and humidity readings logged to `sensor_data.db`
  with timestamps for correlation with network events

## Scope

### In Scope

- LSTM training, inference, and TFLite export pipeline
- MP-QUIC client/server implementation using aioquic over two paths
- Server-side metric monitoring (RTT, goodput) and prediction service
- Proactive path switching execution on Raspberry Pi
- FastAPI backend with REST and WebSocket endpoints
- React dashboard with real-time charts, path status, and history
- SQLite logging for metrics, predictions, and sensor data
- Evaluation scripts comparing LSTM vs Round Robin
- Network degradation emulation using tc netem in lab environment

### Out of Scope

- User authentication or multi-user access control
- Cloud deployment or remote access beyond local LAN
- Support for more than two simultaneous network paths
- Automated LSTM retraining triggered by runtime data
- Mobile-responsive dashboard design
- Support for sensor types other than DHT22
- Production-grade fault tolerance or auto-recovery mechanisms

## Success Criteria

1. LSTM model predicts degradation at least one step ahead with
   accuracy and F1 score reported on the test set from the simulator
2. Path switching is triggered before RTT exceeds the defined
   degradation threshold in at least the majority of test scenarios
3. LSTM-based switching demonstrates lower average RTT and higher
   goodput than Round Robin baseline across the 14 degradation scenarios
4. Real-time dashboard reflects path status and prediction updates
   within 1 second of a network event occurring
5. All evaluation artifacts (charts, CSV logs, metrics) are generated
   automatically by `evaluation/visualize_results.py` without
   manual data processing
6. The full system runs end-to-end on a single LAN: Raspberry Pi →
   server → dashboard, with no external dependencies
